# Java Code Q&A RAG Agent ― 要件定義書（youken_rag.md）

## このファイルの使い方

このファイルは「種」である。Claude Codeに以下のプロンプトを送ることで開発を開始する。

```
youken_rag.md を読んで、指示に従って進めてください。
```

Claude Codeは以下の順序で自動的に進める：

1. **CLAUDE.md を作成** — プロジェクトルールおよびエージェント構成を定義する
2. **SPEC（仕様書）を作成** — 本ファイルの確定要件をもとに詳細仕様を定義する
3. **TDD で実装** — SPEC に従いテストを先行して実装する

---

## 確定要件

本セクションの内容はすべてヒアリング済みの確定事項である。SPECおよび実装はこの要件に完全に従うこと。

### システム概要

JavaプロジェクトのソースコードをベクトルDBにインデックス化し、自然言語による対話型質問応答を提供するRAGエージェントである。対象用途はバグ発見・リファクタリング提案・セキュリティ評価・設計相談・汎用コード質問とする。コードレビューエージェント（`youken.md`）とは別プロジェクトとして実装するが、ファイルスキャン・チャンキング処理は共通化の余地がある。

### 対象コード

- **Javaバージョン：** `config.yaml` の `java_version` パラメータで指定する（デフォルト: 17）。プロンプトテンプレートには `{java_version}` 変数として埋め込むこと
- **入力ソース：** ローカルディレクトリ（コマンド引数で指定）、`src/` 以下を再帰的にスキャンし `.java` ファイルをすべてインデックス化する
- **フレームワーク対応：** 現時点では対応しない。将来的な拡張を想定した設計とすること

### プロジェクト管理

- **複数プロジェクト対応：** 複数のJavaプロジェクトを名前付きで登録・切り替え可能とすること
- **インデックス永続化：** プロジェクトごとにベクトルDBをローカルに永続化すること。デフォルト保存先は `~/.java_qa_agent/indexes/{project_name}/`
- **インデックス再構築：** コード変更時に手動で再構築コマンドを実行する方式とする。差分更新は不要（全件再構築）
- **プロジェクト切り替え：** CLI上でプロジェクト名を指定して切り替える

### LLMバックエンド

- **推論モデル：** `qwen2.5-coder:7b`（Ollama経由）
- **エンベディングモデル：** `nomic-embed-text`（Ollama経由）。推論モデルとエンベディングモデルは分離し、`config.yaml` でそれぞれ独立して設定可能とすること
- **Ollama未起動時の挙動：** 起動時に接続確認を行い、失敗した場合は即時終了してSTDERRにエラーを出力する

### CLIインタフェース

以下のサブコマンドを実装すること：

```
# プロジェクトのインデックスを構築（または再構築）
java-qa index --project <name> --path /path/to/project

# プロジェクトの対話セッションを開始
java-qa chat --project <name>

# 登録済みプロジェクトの一覧を表示
java-qa list

# プロジェクトのインデックスを削除
java-qa delete --project <name>
```

- `chat` サブコマンドは対話ループを維持し、`exit` または `quit` で終了する
- 各ターンで会話履歴を保持し、文脈を引き継いだ質問応答を行うこと（マルチターン対話）
- 会話履歴は直近 `config.yaml` の `max_history_turns` ターン分をコンテキストに含めること

### 出力仕様

- **STDOUT：** 回答テキストをMarkdown形式でSTDOUTに出力する
- **会話ログ：** セッションごとのログを `~/.java_qa_agent/logs/{project_name}/` に保存する（タイムスタンプ付きファイル名）
- **ログ保存の無効化：** `config.yaml` の `save_logs: false` で無効化可能とすること

### 開発ツール・環境

- **開発ツール：** Claude Code を使用して本システムを実装する
- **パッケージマネージャ：** `uv` を使用すること。`pip` の直接使用は容認しない
- **タスクランナー：** `Makefile` を作成し、開発・実行に必要なすべての操作をMakeターゲットとして定義すること
- **ドキュメント：** `README.md` を作成し、セットアップ手順・使い方・Makeターゲット一覧を記載すること

## 絶対必須要件

以下の条件はすべて必須であり、違反は容認しない。

### 1. アーキテクチャ要件

- **LangChain** をRAGチェーンのオーケストレーション層として使用すること
- **ChromaDB** をベクトルDBとして使用すること（ローカル永続化が容易なため）
- **Ollama**（`nomic-embed-text`）をエンベディングバックエンドとして使用すること
- **Ollama**（`qwen2.5-coder:7b`）を推論バックエンドとして使用すること
- 各コンポーネントの責務を明確に分離し、単一責任の原則を守ること
- 将来的なモデル・DBバックエンドの差し替えを想定し、抽象インタフェース経由で呼び出すこと

### 2. 開発手法

- **SPEC駆動開発**を正しく実施すること
- バイブコーディング（場当たり的な実装）は容認しない
- SPECに定義されていない機能をいきなり実装してはならない

### 3. 技術要件

- Python 3.10 以上であること
- **パッケージ管理は `uv` を使用すること。** 依存関係は `pyproject.toml` で管理し、仮想環境の作成・依存解決・スクリプト実行はすべて `uv` 経由で行うこと
- `ruff` によるコーディング規約準拠を維持すること
- 型ヒント（`typing` / `pydantic`）を全関数・クラスに付与すること
- CLIは `typer` を使用して実装すること

### 4. Makefile要件

`Makefile` を作成し、以下のターゲットを**すべて**実装すること。
すべてのコマンドは `uv run` 経由で実行すること。

| ターゲット | 内容 |
|---|---|
| `make install` | 依存関係のインストール（`uv sync`） |
| `make index project=<name> path=<dir>` | 指定プロジェクトのインデックスを構築 |
| `make chat project=<name>` | 指定プロジェクトの対話セッションを開始 |
| `make list` | 登録済みプロジェクトの一覧を表示 |
| `make delete project=<name>` | 指定プロジェクトのインデックスを削除 |
| `make test` | テストスイート全体を実行（`uv run pytest`） |
| `make lint` | `ruff check` によるlint実行 |
| `make format` | `ruff format` によるフォーマット実行 |
| `make typecheck` | `mypy` による型チェック実行 |
| `make clean` | `__pycache__` 等の生成物を削除 |
| `make help` | 利用可能なターゲットの一覧と説明を表示 |

### 5. README.md要件

`README.md` を作成し、以下の内容をすべて記載すること。

- **プロジェクト概要** — 何をするツールか（1〜2段落）
- **前提条件** — Python・uv・Ollama・必要モデルのインストール手順（コマンド付き）
  - `ollama pull qwen2.5-coder:7b`
  - `ollama pull nomic-embed-text`
- **セットアップ手順** — リポジトリクローンから実行可能状態までの手順（コマンド付き）
- **使い方** — 基本的な実行例（コマンド付き）。以下を含むこと：
  - インデックス構築例
  - 対話セッション開始例
  - プロジェクト切り替え例
- **Makeターゲット一覧** — 全ターゲットの説明表
- **設定** — `config.yaml` の主要パラメータ説明
- **プロジェクト構成** — ディレクトリツリーと各ファイルの役割

### 6. RAGエンジン要件

- **チャンキング：** `.java` ファイルをメソッド単位に分割してインデックス化すること。1,000トークン未満のファイルはファイル単位でインデックス化する
- **コンテキスト付与：** 各チャンクにインポート宣言・クラスシグネチャ・メンバ変数をメタデータとして付与すること
- **Retrieval：** 質問文に対して類似度上位 `top_k`（デフォルト: 5）件のチャンクを取得すること。`top_k` は `config.yaml` で設定可能とすること
- **マルチターン対話：** 会話履歴を保持し、前の質問の文脈を引き継いだ回答を生成すること
- **トークン制限：** 取得チャンク + 会話履歴 + 質問文の合計が `max_input_tokens`（デフォルト: 3,000）を超えないよう制御すること

### 7. リソース最適化

- **Concurrency Control：** Ollamaへの推論リクエストはシリアル実行（Max Concurrency: 1）とすること。根拠はコードレビューエージェントと同様（6GB VRAM制約）
- **エンベディングのバッチ処理：** インデックス構築時のエンベディングはバッチで処理し、Ollamaへのリクエスト回数を最小化すること

### 8. エラーハンドリング

- **Ollama接続失敗：** 起動時の接続確認で失敗した場合は即時終了し、STDERRにエラー種別・エンドポイント・推奨対処手順を出力すること
- **モデル未取得：** 推論モデル・エンベディングモデルのいずれかがOllamaに存在しない場合、STDERRに必要な `ollama pull` コマンドを明示してエラー終了すること
- **インデックス未構築：** `chat` 実行時にプロジェクトのインデックスが存在しない場合、エラーメッセージとともに `make index` コマンドの実行を促すこと
- **存在しないプロジェクト：** 未登録のプロジェクト名が指定された場合、登録済みプロジェクト一覧を表示してエラー終了すること
- **LLM出力パースエラー：** `try/except` で保護し、パース失敗時はエラー内容をSTDERRに出力してセッションを継続すること

### 9. テスト駆動開発（TDD）

テストを先に書き、テストが通るように実装すること。以下のカテゴリをすべてカバーすること：

- **単体テスト** — Indexer・Retriever・各コンポーネントの入出力が仕様通りか
- **チャンキングテスト** — メソッド単位分割・メタデータ付与が正しく機能するか
- **スキーマテスト** — Pydanticモデルのバリデーションが正しく機能するか
- **RAGチェーンテスト** — Retrieval→生成の一連の流れが正しく動作するか（モック可）
- **マルチターンテスト** — 会話履歴が正しく引き継がれるか
- **プロジェクト管理テスト** — 登録・切り替え・削除が正しく動作するか
- **エラーハンドリングテスト** — Ollama未起動・モデル未取得・インデックス未構築・未登録プロジェクトへの対処
- **トークン制限テスト** — `max_input_tokens` 超過時に正しく切り詰められるか
- **エッジケーステスト** — 空プロジェクト・`.java` ファイルなし・極端に大きいファイル

## システム構成

### RAGパイプライン

```
[インデックス構築時: make index project=<name> path=<dir>]

[FileScanner] ── src/ 以下の .java ファイルを再帰スキャン
       │
       ▼
[Chunker] ── メソッド単位に分割
             インポート・クラスシグネチャ・メンバ変数をメタデータとして付与
       │
       ▼
[Embedder] ── nomic-embed-text でベクトル化（バッチ処理）
       │
       ▼
[ChromaDB] ── ~/.java_qa_agent/indexes/{project_name}/ に永続化


[質問応答時: make chat project=<name>]

[ユーザー入力（質問文）]
       │
       ▼
[Embedder] ── 質問文をベクトル化
       │
       ▼
[Retriever] ── ChromaDB から類似チャンク上位 top_k 件を取得
       │
       ▼
[Context Builder] ── 取得チャンク + 会話履歴 + 質問文を結合
                      max_input_tokens を超えないよう制御
       │
       ▼
[Ollama（qwen2.5-coder:7b）] ── 回答生成
       │
       ▼
[STDOUT（Markdown）] + [ログ保存]
       │
       ▼
[次の質問へ（マルチターンループ）]
```

## CLAUDE.md に含めるべきルール

### 開発ルール

- SPEC（仕様書）を作成し、それに従って全体を実装すること
- テスト駆動開発（TDD）で進めること。テストを先に書き、テストが通るように実装する
- 型ヒントを全コードに付与すること
- パッケージ管理は `uv` を使用すること。`pip` の直接使用は容認しない
- エラーが出たら自分でデバッグすること
- 要件に関して不明点がある場合は、必ずユーザーに質問すること
- 日本語で応答すること

### SPEC修正ルール（バイブコーディング防止）

ユーザーから修正指示があった場合、いきなりコードを修正してはならない。
必ず以下の手順を踏むこと：

1. まず **SPEC を修正する**
2. 修正した SPEC をユーザーに提示し、「SPECシートを修正しました。確認してください。OKであれば実装します。」と確認する
3. ユーザーからOKを得てから、SPECに従って実装する

この手順の省略は容認しない。

### 環境変数管理ルール

- 認証情報は `.env` ファイルで管理し、`python-dotenv` で読み込むこと
- `.env` は `.gitignore` に追加し、リポジトリにコミットしないこと
- `.env.example` を作成し、必要な変数名と説明を記載すること

### ステアリング（方向維持）

- この CLAUDE.md の内容をステアリングファイルとして、最後まで参照し続けること
- 要件から外れていないか、実装の節目ごとに自己チェックすること
- コンテキストが長くなったら自分で整理すること

## SPEC（仕様書）の作成ルール

### 品質基準

- 本 `youken_rag.md` の確定要件をすべてSPECに反映すること
- 曖昧な部分を残さない
- SPECに書いていないことは実装しない。SPECに書いてあることは必ず実装する

### ファイル構成

- `spec/architecture.md` — 全体アーキテクチャ、コンポーネント役割分担、データフロー
- `spec/components.md` — 各コンポーネントの詳細仕様（入力・処理・出力・エラー処理）
- `spec/prompts.md` — プロンプトテンプレート（`{java_version}`・`{context}`・`{history}`・`{question}` 変数を含む）
- `spec/schemas.md` — Pydanticモデル定義、入出力スキーマ
- `spec/cli.md` — CLIサブコマンドの詳細仕様（引数・オプション・出力・エラー）
- `spec/test-plan.md` — テスト計画、テストケース一覧

### SPECに含めるべき項目

1. **システム概要**
2. **CLIサブコマンド詳細仕様**（引数・オプション・正常系・異常系の挙動）
3. **インデックス構築仕様**（チャンキングアルゴリズム・メタデータ付与・バッチエンベディング）
4. **RAG検索仕様**（類似度計算方法・`top_k` 制御・トークン上限制御）
5. **マルチターン対話仕様**（履歴管理・履歴のトークン上限制御）
6. **プロジェクト管理仕様**（登録・切り替え・削除・一覧表示の内部データ構造）
7. **エラーハンドリング仕様**
8. **設定管理仕様（`config.yaml` の全パラメータ定義）**

   ```yaml
   java_version: 17
   ollama:
     base_url: "http://localhost:11434"
     model: "qwen2.5-coder:7b"
     embed_model: "nomic-embed-text"
     timeout_seconds: 120
   rag:
     top_k: 5
     chunk_token_threshold: 1000
     max_input_tokens: 3000
     max_history_turns: 6
   storage:
     index_dir: "~/.java_qa_agent/indexes"
     log_dir: "~/.java_qa_agent/logs"
     save_logs: true
   ```

9. **機能一覧（優先度付き）**（P0 / P1 / P2）

## 注意事項

- `pip` を直接使用することは容認しない。すべてのパッケージ操作は `uv` 経由で行うこと
- `nomic-embed-text` モデルは事前に `ollama pull nomic-embed-text` で取得が必要である。起動時に推論モデルとエンベディングモデルの両方の存在確認を行い、未取得の場合はSTDERRに `ollama pull` コマンドを明示してエラー終了すること
- ChromaDBの永続化ディレクトリが存在しない場合は自動作成すること
- インデックス再構築は全件削除・全件再挿入とする。部分更新は実装しない
- 将来的なフレームワーク対応・Javaバージョン固有ルールの追加を見越し、プロンプトテンプレートは `config.yaml` または独立したモジュールとして外部化すること
- コードレビューエージェント（`java-review-agent`）とファイルスキャン・チャンキングのロジックが重複する場合、将来的な共通ライブラリ化を念頭に置いた設計とすること
