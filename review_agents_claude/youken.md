# Java Code Review AI Agent ― 要件定義書（youken.md）

## このファイルの使い方

このファイルは「種」である。Claude Codeに以下のプロンプトを送ることで開発を開始する。

```
youken.md を読んで、指示に従って進めてください。
```

Claude Codeは以下の順序で自動的に進める：

1. **CLAUDE.md を作成** — プロジェクトルールおよびエージェント構成を定義する
2. **SPEC（仕様書）を作成** — 本ファイルの確定要件をもとに詳細仕様を定義する
3. **TDD で実装** — SPEC に従いテストを先行して実装する

---

## 確定要件

本セクションの内容はすべてヒアリング済みの確定事項である。SPECおよび実装はこの要件に完全に従うこと。

### 対象コード

- **Javaバージョン：** `config.yaml` の `java_version` パラメータで指定する（デフォルト: 17）。将来的な他バージョンへの対応を想定し、各エージェントのプロンプトテンプレートには `{java_version}` 変数として埋め込むこと
- **入力ソース：** ローカルディレクトリ（実行時引数で指定）、`src/` 以下を再帰的にスキャンし `.java` ファイルをすべて処理する
- **フレームワーク対応：** 現時点では対応しない。ただし将来的なフレームワーク固有ルール追加を想定し、拡張可能な設計とすること

### レビュー観点と優先度

以下の優先度順でレビューを行う。優先度はAggregatorノードでの重み付けおよびレポート内の表示順に反映すること。

| 優先度 | 観点 | 備考 |
|---|---|---|
| 1 | バグ検出 | ロジックエラー、NullPointerException、リソースリーク等 |
| 2 | セキュリティ | インジェクション、不適切な認証・認可、機密情報の露出等 |
| 3 | 効率性 | アルゴリズム・データ構造の非効率、不要なオブジェクト生成等 |
| 4 | 設計 | SOLID原則違反、不適切なパターン適用等 |
| 5 | 可読性 | コーディング規約は**大まかな指摘にとどめる**。細かい規約違反の網羅的列挙は行わない |

### LLMバックエンド

- **使用モデル：** `qwen2.5-coder:7b`（Ollama経由）
- **全ノード共通：** 単一モデルを全エージェントノードで使用する
- **Ollama未起動時の挙動：** 起動時に接続確認を行い、失敗した場合は即時終了してSTDERRにエラーを出力する

### 出力仕様

- **ファイル単位レポート：** 各 `.java` ファイルに対して個別のMarkdownレポートを生成する
- **全体サマリー：** 全ファイル処理完了後、プログラム全体の流れ（モジュール間の依存関係・データフロー・アーキテクチャ上の推奨事項）を評価したサマリーレポートを生成する。**レビューがスキップされたファイル/メソッドの一覧を必ずサマリーに含めること**
- **出力先（ファイル）：** `./review_output/` ディレクトリ（固定）
- **出力先（標準出力）：** STDOUTにも同内容を出力する
- **出力形式：** Markdownのみ

### 開発ツール

- **Claude Code** を使用して本システムを実装する

## 絶対必須要件

以下の条件はすべて必須であり、違反は容認しない。

### 1. アーキテクチャ要件

- **LangChain / LangGraph** をオーケストレーション層として使用すること
- **Ollama**（`qwen2.5-coder:7b`）をローカルLLM推論バックエンドとして使用すること
- 各エージェントの責務を明確に分離し、単一責任の原則を守ること
- 将来的なフレームワーク固有ルールおよびJavaバージョン固有ルールの追加を想定し、エージェントノードおよびルールセットを差し替え・追加可能な設計とすること

### 2. 開発手法

- **SPEC駆動開発**を正しく実施すること
- バイブコーディング（場当たり的な実装）は容認しない
- SPECに定義されていない機能をいきなり実装してはならない

### 3. 技術要件

- Python 3.10 以上であること
- 依存関係は `pyproject.toml` で管理すること
- `ruff` によるコーディング規約準拠を維持すること
- 型ヒント（`typing` / `pydantic`）を全関数・クラスに付与すること

### 4. レビューエンジン要件

- レビュー観点ごとに独立したエージェントノードを定義すること
- 各ノードの出力スキーマを `pydantic` v2 モデルで定義すること
- 複数エージェントの結果を集約するAggregatorノードを設けること
- Aggregatorは確定要件の優先度順に従い、重み付けおよび重複除去を行うこと
- 最終出力はMarkdown形式とすること

### 5. リソース最適化と分割処理

#### Concurrency Control

- Ollamaへの推論リクエストは**シリアル実行（Max Concurrency: 1）**をデフォルトとすること
- 根拠：`qwen2.5-coder:7b` が約4.7GB VRAMを占有するため、6GB VRAM環境では並列推論リクエストによるOOMリスクを排除する必要がある
- `config.yaml` の `max_concurrency` パラメータでオーバーライド可能とすること

#### Logical Chunking

- `Preprocessor` ノードにて、対象Javaファイルのトークン数を推定し、**1,000トークンを超える場合はメソッド単位に分割**して各エージェントに渡すこと
- 分割した各スロットには以下のコンテキストを必ず付与すること：
  - インポート宣言（`import` 文全体）
  - クラスシグネチャ（クラス名・継承・実装インタフェース）
  - クラスのメンバ変数および型情報
- 分割閾値（デフォルト: 1,000トークン）は `config.yaml` でオーバーライド可能とすること

#### Token Management

- 各エージェントに渡すコンテキストは**最大3,000トークン**に制限すること
- 応答用リザーブ領域として**1,000トークン**を確保すること
- これらの値は `config.yaml` の `max_input_tokens` / `response_reserve_tokens` パラメータでオーバーライド可能とすること
- 備考：`qwen2.5-coder:7b` のコンテキストウィンドウは32,768トークンであるため、この制限は推論速度の現実的な確保を目的とした設定である

### 6. エラーハンドリング

#### Ollama接続失敗

- 起動時の接続確認で失敗した場合は即時終了し、STDERRに以下の情報を出力すること：エラー種別・Ollamaのエンドポイント・推奨対処手順

#### OOM / Timeout Fallback

- ローカルモデルがリソース不足またはタイムアウトでエラーを返した場合、対象ファイル（またはメソッドスロット）のレビューを `Skipped (Resource Limit)` として記録し、**スキャン全体を停止させないこと**
- スキップされた対象は構造化データとして保持し、最終サマリーレポートに一覧として出力すること

#### LLM出力パースエラー

- LLMの出力パースは `try/except` で保護し、パース失敗時は `Skipped (Parse Error)` として構造化エラーをレポートに記録すること
- パースエラーも最終サマリーのスキップ一覧に含めること

### 7. テスト駆動開発（TDD）

テストを先に書き、テストが通るように実装すること。以下のカテゴリをすべてカバーすること：

- **単体テスト** — 各エージェントノードの入出力が仕様通りか
- **スキーマテスト** — Pydanticモデルのバリデーションが正しく機能するか
- **グラフ遷移テスト** — LangGraphの状態遷移が期待通りに進むか
- **統合テスト** — Ollamaとの通信が成立するか（モック可）
- **チャンキングテスト** — 1,000トークン超のファイルが正しくメソッド単位に分割されるか、コンテキストが正しく付与されるか
- **トークン制限テスト** — 3,000トークン超のコンテキストが正しく切り詰められるか
- **エラーハンドリングテスト** — Ollama未起動・OOM・タイムアウト・パースエラーへの対処
- **回帰テスト** — 既知のバグパターン・セキュリティ脆弱性パターンに対してレビューが正しく機能するか
- **エッジケーステスト** — 空ファイル・巨大ファイル・構文エラーを含むコード・`src/` 以下に `.java` ファイルが存在しない場合

## エージェント構成

### 役割分担

| エージェント | 役割 | 実行方式 |
|---|---|---|
| **Orchestrator（LangGraph）** | グラフ全体の状態管理・ルーティング・ファイルスキャン | Python プロセス |
| **Preprocessor** | コード読み込み・トークン数推定・メソッド単位チャンキング・メタデータ抽出 | Python |
| **Bug Detector** | バグ・ロジックエラーの検出（優先度1） | Ollama（`qwen2.5-coder:7b`） |
| **Security Scanner** | セキュリティ脆弱性の検出（優先度2） | Ollama（`qwen2.5-coder:7b`） |
| **Efficiency Analyzer** | アルゴリズム・I/O効率の分析（優先度3） | Ollama（`qwen2.5-coder:7b`） |
| **Design Critic** | 設計パターン・SOLID原則の評価（優先度4） | Ollama（`qwen2.5-coder:7b`） |
| **Style Reviewer** | 可読性・コーディング規約の大まかな評価（優先度5） | Ollama（`qwen2.5-coder:7b`） |
| **Aggregator** | 全エージェント結果の統合・優先度付け・重複除去 | Python（ルールベース） |
| **File Report Generator** | ファイル単位Markdownレポートの生成 | Pythonテンプレート |
| **Summary Generator** | プロジェクト全体サマリー（スキップ一覧含む）の生成 | Ollama（`qwen2.5-coder:7b`） |

### LangGraph グラフ構造

```
[入力: プロジェクトディレクトリ]
       │
       ▼
[FileScanner] ── src/ 以下の .java ファイルを再帰スキャン
       │
       ▼（ファイルごとにループ）
[Preprocessor] ── トークン数推定 → 1,000トークン超の場合メソッド単位に分割
                   インポート・クラスシグネチャ・メンバ変数を各スロットに付与
       │
       ▼（スロットごと、シリアル実行）
       ├──→ [Bug Detector]
       ├──→ [Security Scanner]
       ├──→ [Efficiency Analyzer]
       ├──→ [Design Critic]
       └──→ [Style Reviewer]
              │（各スロット完了後に集約）
              ▼
       [Aggregator] ── 優先度順に統合・重複除去
              │
              ▼
       [File Report Generator] ── ./review_output/{filename}.md に保存 + STDOUT出力
              │
              ▼（全ファイル処理完了後）
       [Summary Generator] ── プロジェクト全体サマリー + スキップ一覧を生成
              │
              ▼
       [出力: ./review_output/summary.md + STDOUT]
```

備考：Concurrency Control要件により、各エージェントへの推論リクエストはシリアルで実行する。LangGraphの並列Send APIは使用しない。

## CLAUDE.md に含めるべきルール

CLAUDE.md を作成する際、以下のルールを必ず含めること。

### 開発ルール

- SPEC（仕様書）を作成し、それに従って全体を実装すること
- テスト駆動開発（TDD）で進めること。テストを先に書き、テストが通るように実装する
- 型ヒントを全コードに付与すること
- エラーが出たら自分でデバッグすること
- 要件に関して不明点がある場合は、必ずユーザーに質問すること
- 日本語で応答すること

### SPEC修正ルール（バイブコーディング防止）

ユーザーから修正指示があった場合、いきなりコードを修正してはならない。
必ず以下の手順を踏むこと：

1. まず **SPEC を修正する**
2. 修正した SPEC をユーザーに提示し、「SPECシートを修正しました。確認してください。OKであれば実装します。」と確認する
3. ユーザーからOKを得てから、SPECに従って実装する

この手順の省略は容認しない。

### 環境変数管理ルール

- 認証情報は `.env` ファイルで管理し、`python-dotenv` で読み込むこと
- `.env` は `.gitignore` に追加し、リポジトリにコミットしないこと
- `.env.example` を作成し、必要な変数名と説明を記載すること

### ステアリング（方向維持）

- この CLAUDE.md の内容をステアリングファイルとして、最後まで参照し続けること
- 要件から外れていないか、実装の節目ごとに自己チェックすること
- コンテキストが長くなったら自分で整理すること

## SPEC（仕様書）の作成ルール

### 品質基準

- 本 `youken.md` の確定要件をすべてSPECに反映すること。要件をはしょることは容認しない
- 曖昧な部分を残さない。すべてのエージェント・ノードについて、入出力スキーマ・処理内容・エラー処理を具体的に定義すること
- SPECに書いていないことは実装しない。SPECに書いてあることは必ず実装する

### ファイル構成

SPECは以下のように複数ファイルに分割すること：

- `spec/architecture.md` — 全体アーキテクチャ、エージェント役割分担、データフロー
- `spec/agents.md` — 各エージェントノードの詳細仕様（入力・処理・出力・エラー処理）
- `spec/prompts.md` — 各LLMエージェントに渡すプロンプトテンプレート（`{java_version}` 変数を含む）
- `spec/schemas.md` — Pydanticモデル定義、入出力スキーマ
- `spec/test-plan.md` — テスト計画、テストケース一覧

### SPECに含めるべき項目

1. **システム概要** — 何を行うシステムか（詳細に）
2. **入出力仕様** — 入力形式（ディレクトリパス）、出力形式（Markdown）、出力先
3. **エージェント詳細仕様**
   - 各ノードの名称・役割・使用LLMバックエンド
   - 入力スキーマ（Pydanticモデル）
   - 出力スキーマ（Pydanticモデル）
   - プロンプトテンプレート（`{java_version}` を含む変数一覧・フォーマット）
   - タイムアウト・リトライ設定
4. **Preprocessorの分割処理仕様**
   - トークン数推定方法
   - メソッド単位分割のアルゴリズム
   - 各スロットへのコンテキスト付与仕様（インポート・クラスシグネチャ・メンバ変数）
5. **グラフ遷移仕様**
   - LangGraphの状態定義（`TypedDict` または `BaseModel`）
   - ノード間の遷移条件（条件付きエッジを含む）
   - シリアル実行の制御方法
6. **エラーハンドリング仕様**
   - Ollama接続失敗時：即時終了・STDERRへの出力内容
   - OOM/Timeout時：`Skipped (Resource Limit)` 記録・スキャン継続
   - LLM出力パースエラー時：`Skipped (Parse Error)` 記録
   - スキップ一覧のサマリーへの出力仕様
7. **設定管理仕様（`config.yaml` の全パラメータ定義）**

   ```yaml
   java_version: 17
   ollama:
     base_url: "http://localhost:11434"
     model: "qwen2.5-coder:7b"
     timeout_seconds: 120
   processing:
     max_concurrency: 1
     chunk_token_threshold: 1000
     max_input_tokens: 3000
     response_reserve_tokens: 1000
   output:
     dir: "./review_output"
   ```

8. **機能一覧（優先度付き）**
   - P0（必須）：最低限動作するために必要な機能
   - P1（重要）：あると良い機能
   - P2（任意）：余裕があれば実装する機能

## 注意事項

- Ollama が未起動の場合、起動時の接続確認で即時終了しSTDERRにエラーを出力すること。起動後の途中切断はリトライ後に `Skipped (Resource Limit)` として記録しスキャンを継続すること
- LLMの出力は非決定的であるため、出力パースは `try/except` で保護し、パース失敗時は `Skipped (Parse Error)` としてレポートに記録すること
- `review_output/` ディレクトリが存在しない場合は実行時に自動作成すること
- 将来的なフレームワーク対応・Javaバージョン固有ルールの追加を見越し、プロンプトテンプレートおよびルールセットは `config.yaml` または独立したモジュールとして外部化すること
